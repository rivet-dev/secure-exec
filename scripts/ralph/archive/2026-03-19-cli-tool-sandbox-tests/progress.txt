## Codebase Patterns

- All sandbox tests use `createTestNodeRuntime()` or `NodeRuntime` directly, then `proc.exec()` / `proc.run()` to execute code inside the VM
- Look at `tests/test-suite/node.test.ts` for the canonical pattern of creating a runtime and running sandbox code
- Mock LLM server at `tests/cli-tools/mock-llm-server.ts` is fine running on host — sandbox code reaches it via network bridge
- Pi has `createAgentSession()` programmatic API in `@mariozechner/pi-coding-agent` (dist/index.js) — designed for in-process headless use
- Claude Code SDK always spawns cli.js as subprocess (ProcessTransport class) — no in-process execution path
- OpenCode npm package ships only a compiled Bun binary (ELF) — no JS source to load in VM
- Interactive tests should use `kernel.openShell()` with `@xterm/headless`, not host PTY via `script -qefc`
- **ModuleAccessFileSystem** maps sandbox `/root/node_modules/` to host `<cwd>/node_modules/` — use `moduleAccess: { cwd: PACKAGE_ROOT }` to point at the right host node_modules
- **Sandbox referrer path** must be a sandbox-internal path like `/root/_sandbox_entry_.mjs`, NOT a host filesystem path — the module resolver uses this to walk up finding `/root/node_modules/`
- **Pi's ESM module graph is too complex for isolated-vm** — `await import('@mariozechner/pi-coding-agent')` hangs due to the large dependency tree. Tests skip with clear reason. This will resolve when the bridge supports full ESM loading.
- Use `createDefaultNetworkAdapter({ initialExemptPorts: new Set([port]) })` to exempt mock server ports from SSRF protection
- Use `allowAll` from `shared/permissions.ts` for full sandbox access (fs, network, child_process, env)
- `CommandExecutor` interface needs a real host implementation wrapping `node:child_process.spawn` for the child_process bridge to work
