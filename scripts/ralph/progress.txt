## Codebase Patterns
- Run upstream Node bootstrap probes in isolated child host Node processes with `--expose-internals`; `internal/bootstrap/node` can abort on native asserts and kill the current test worker if run inline.
- For bootstrap preflight work, read builtin source from `process.binding('natives')` and defer pinned vendored asset sync to the dedicated upstream-asset story instead of partially vendoring bootstrap files early.
- WasmVM socket tests that create client sockets from the test harness must register a real kernel-owned PID first; fake PID literals like `999` are rejected by `SocketTable`.
- When WasmVM `host_net` imports change, rebuild `@secure-exec/wasmvm` before validating probes; runtime execution uses `packages/wasmvm/dist/kernel-worker.js`, not just `src/kernel-worker.ts`.
- Upstream Node asset sync must export from a pinned git ref and validate with `pnpm check:node-upstream-assets`; do not copy builtin files from a dirty checkout or from upstream `main` when the story is pinned to a release tag.
- `pnpm check:node-upstream-assets` only needs a git checkout that contains the pinned commit; a temporary `git init` plus single-commit fetch is enough for real-release validation.
- Resolve vendored upstream Node assets relative to `import.meta.url` from `packages/nodejs/src/upstream/` so the same loader works under source tests and built `dist/` output.
- Upstream builtin `compileFunction()` must mirror Node's wrapper categories from `src/builtin_info.h`: `internal/bootstrap/realm` uses the realm wrapper, other `internal/bootstrap/*` and `internal/main/*` use `(process, require, internalBinding, primordials)`, `internal/per_context/*` uses `(exports, primordials, privateSymbols, perIsolateSymbols)`, and other builtins use the CommonJS-style wrapper.
- Snapshot-free upstream bootstrap bring-up should run vendored `internal/*` assets in a dedicated `node --expose-internals` child that captures stdout/stderr into structured JSON, while temporary public-builtin fallback stays explicit and limited to host `node:` modules touched by the `eval_string` smoke path.
- Model upstream internal bindings with separate fields for execution model (`host-only` vs `host-lifecycle-plus-backend` vs `deferred`), implementation status, and required phase (`bootstrap` vs `fs-first`) so later stories can wire providers incrementally without losing blocker tracking.
- For replacement-runtime binding gaps, prefer explicit registry entries that fail with phase-aware notes over implicit missing bindings; that keeps `fs`/loader blockers reviewable before the real provider exists.
- Keep public-builtin first-light bring-up opt-in: vendoring just the target public builtin (for example `fs`) while other public builtins still fall back to host `node:` modules is enough to prove the new loader path without broadening scope prematurely.
- Async isolated-child upstream probes should use an explicit completion signal such as `process.__secureExecDone()` instead of guessing from active handles or requests; bootstrap can leave long-lived handles around that make idle detection unreliable.
- Public replacement-runtime wrappers should vendor `fs` by default but only wait for `process.__secureExecDone()` when the evaluated source actually uses that handshake; hardcoded always-wait mode deadlocks plain `node -e` smoke paths.
- Host-backed stats that flow into Wasm filestat packing must truncate Node's fractional `*.atimeMs`/`*.mtimeMs`/`*.ctimeMs`/`*.birthtimeMs` values to integer milliseconds before any `BigInt(...)` conversion.
- Vendored upstream bootstrap callback fs first-light is stable when backend work runs synchronously and `FSReqCallback.oncomplete` is deferred with `queueMicrotask()`; relying on helper-process async close events can leave the isolated runner waiting forever.
- Vendored `fs.promises` and path-based stream bring-up still imports `internal/fs/watchers`; when the bootstrap runner reuses host `internalBinding('fs_event_wrap')`, wrap `FSEvent` in a fresh subclass first or the vendored watcher module will throw when it redefines the legacy `owner` alias on the prototype.
- Vendored async `net` bring-up in the exposed-internals bootstrap runner needs two special cases: subclass host `tcp_wrap.TCP` before `lib/net.js` defines its legacy `owner` alias, and execute `process.__secureExecDone()` payloads in a post-bootstrap CommonJS wrapper that can hand `require('net')` a host-context compiled vendored module; leaving async `net` inside the reduced `internal/main/eval_string` path exits before socket callbacks drain.
- Vendored async `http` / `https` first-light in the exposed-internals bootstrap runner must host-context compile the whole HTTP family together, including vendored `internal/http`; mixing pinned `http.js` with host `_http_*` helpers or host `internal/http` breaks APIs like `getGlobalAgent()` and stalls HTTP round-trips.
- Nested upstream bootstrap children must strip inherited `NODE_CHANNEL_FD` / `NODE_UNIQUE_ID` and use an explicit env flag for live-stdio IPC mode; `process.send` alone is not a reliable signal under forked test workers.
