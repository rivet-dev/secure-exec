## Codebase Patterns
- CommandRegistry is in packages/kernel/src/command-registry.ts, tests in test/command-registry.test.ts
- TestFileSystem from test/helpers.ts is used for VFS tests (import dynamically in test)
- createMockDriver helper is defined locally in the test file (not from helpers.ts)
- Pre-existing TS error in resource-exhaustion.test.ts (TS18047) and flaky PTY stress test — both on main
- The kernel package typecheck script is `pnpm run typecheck` (not turbo `check-types`)
- MockRuntimeDriver in test/helpers.ts has an optional `tryResolve` property for on-demand discovery tests
- spawnInternal extracts basename before calling tryResolve — drivers always receive the bare command name
- populateBinEntry is async but called fire-and-forget from spawnInternal (sync context); tests need a microtask yield to observe the result
- Standalone binary crate naming: libs use `secureexec-*` (e.g. `secureexec-grep`), command crates use `cmd-*` (e.g. `cmd-grep`)
- Library crates go in `wasmvm/crates/libs/`, binary crates in `wasmvm/crates/commands/`
- Binary crates use `[[bin]] name = "command-name"` and depend on the corresponding library crate
- `cargo build -p cmd-grep --target wasm32-wasip1 -Z build-std=std,panic_abort --release` for WASM builds
- Standalone grep binary is ~864K vs ~15MB multicall — significant size reduction
- WasmVM driver typecheck: `cd packages/runtime/wasmvm && pnpm run typecheck`
- WasmVM driver tests: `cd packages/runtime/wasmvm && pnpm run test`
- Use `node:fs/promises` `open()` (returns FileHandle) for async file operations, NOT separate `read`/`close` imports
- `isWasmBinary()` (async) and `isWasmBinarySync()` are in `packages/runtime/wasmvm/src/wasm-magic.ts`
- Driver in commandDirs mode: commands discovered from filesystem scan, _commandPaths maps name→path
- Driver in legacy mode (wasmBinaryPath only): uses static WASMVM_COMMANDS list, all commands use single binary
- Permission enforcement is in kernel-worker.ts (WASI host imports), not the driver — use `permission-check.ts` helpers
- `PermissionTier` type is in syscall-rpc.ts; driver resolves tier via `_resolvePermissionTier(command)`
- Permission system is opt-in: no `permissions` config → all commands `full` (unrestricted); with config, resolution is: user exact > user glob > user `*` > DEFAULT_FIRST_PARTY_TIERS > 'read-write'
- `resolvePermissionTier()` in permission-check.ts supports a `defaults` fallback layer; `DEFAULT_FIRST_PARTY_TIERS` in driver.ts maps known first-party commands to sensible tiers
- proc_spawn enforcement: only `full` tier allows subprocess spawning (isSpawnBlocked helper)
- Multicall crate is deleted; all commands are standalone binaries in `crates/commands/`
- Tests use `commandDirs: [COMMANDS_DIR]` where `COMMANDS_DIR = wasmvm/target/wasm32-wasip1/release/commands`
- CI guard test variable is `hasWasmBinaries` (checks for commands dir, not multicall binary)
- Legacy `wasmBinaryPath` option still works (deprecated) but tests now use commandDirs for real WASM execution
- C build system: `wasmvm/c/Makefile` handles wasi-sdk download, C→WASM compilation, and install
- Top-level `make c-wasm` (in wasmvm/) builds C programs; `make -C wasmvm/c programs` directly
- wasi-sdk cached in `wasmvm/c/vendor/wasi-sdk/`; vanilla sysroot at `vendor/wasi-sdk/share/wasi-sysroot`
- C WASM output goes to `wasmvm/c/build/`; native test binaries to `wasmvm/c/build/native/`
- Per-program Makefile overrides for multi-file compilation (cJSON, SQLite): define `$(BUILD_DIR)/prog:` target
- C parity test: `commandDirs: [C_BUILD_DIR, COMMANDS_DIR]` — C first for precedence, COMMANDS_DIR for `sh`
- C parity test skip: check `existsSync` for hello binary in all 3 dirs (COMMANDS_DIR, C_BUILD_DIR, NATIVE_DIR)
- wasi-libc pinned commit (wasi-sdk-25): `574b88da481569b65a237cb80daf9a2d5aeaf82d`; cloned to `c/vendor/wasi-libc/`
- Patched sysroot: `make -C wasmvm/c sysroot`; output at `c/sysroot/`; verify via `c/sysroot/lib/wasm32-wasi/libc.a`
- wasi-libc patches go in `wasmvm/patches/wasi-libc/`; applied by `scripts/patch-wasi-libc.sh`
- Must `make clean` in wasi-libc before rebuild — stale objects break `check-symbols` target
- Canonical WASM import signatures: `wasmvm/crates/wasi-ext/src/lib.rs` — C patches must match exactly
- Patch `@@ -X,OLD +X,NEW @@` counts must be exact: context lines count in both OLD and NEW, `-` only in OLD, `+` only in NEW
- musl's `process/` dir NOT compiled for WASI — reimplements (posix_spawn, waitpid, file_actions helpers) go in `libc-bottom-half/sources/`
- posix_spawn ABI: argv/envp serialized as null-separated strings; first argv entry = command; file_actions → stdin/stdout/stderr fd params
- wasi-libc `libc-bottom-half/sources/` files: use `<__errno.h>` NOT `<errno.h>` (avoids musl's `features.h` `weak` macro)
- For errno constants (EBADF etc.), include `<__errno_values.h>` separately from `<__errno.h>`
- `<spawn.h>` and `<sys/wait.h>` are NOT in wasi sysroot — define needed types inline in source files
- signal.h's `__NEED_pid_t`+`bits/alltypes.h` are inside `#ifdef __wasilibc_unmodified_upstream` — add your own outside when exposing `pid_t`-using declarations
- wasi-libc's `sigset_t` is `unsigned char` (not `unsigned long`) — use `__NEED_sigset_t`+`bits/alltypes.h` for correct type
- Patched sysroot: `include/wasm32-wasip1` and `lib/wasm32-wasip1` are symlinks to `wasm32-wasi` (clang --target=wasm32-wasip1 expects wasip1 paths)
- wasi-libc build uses `libc` target only (not `finish`) to skip `check-symbols` validation that rejects custom __host_* symbols
- CRT objects (crt1.o) are copied from vanilla wasi-sdk sysroot — standard startup code, no patches needed
- Programs requiring patched sysroot: getpid_test, userinfo, pipe_test, dup_test (listed in Makefile `PATCHED_PROGRAMS`)
- Parity tests: exact match for deterministic output; structural/format match where values inherently differ (PID, uid/gid, timing)
- WASM C programs using posix_spawn/waitpid: `#ifdef __wasi__` with inline type declarations (spawn.h/sys/wait.h not in wasi sysroot)
- posix_spawn_file_actions_t WASM layout: `{ int __pad0[2]; void *__actions; int __pad[16]; }` — must match host_spawn_wait.c

## 2026-03-19 4:40 PM PDT - US-033
- Vendored SQLite amalgamation (v3.46.1, public domain) into `wasmvm/c/libs/sqlite3/`
- Created `wasmvm/c/programs/sqlite3_mem.c` — in-memory DB: create table, insert 4 rows, SELECT all/AVG/TOP queries
  - Uses `#ifdef SQLITE_OS_OTHER` guard for WASM-only `sqlite3_os_init()`/`sqlite3_os_end()` stubs
  - Native build uses default Unix VFS (no SQLITE_OS_OTHER), WASM uses SQLITE_OS_OTHER for in-memory only
- Updated `wasmvm/c/Makefile` with SQLite compilation rules:
  - WASM: `-DSQLITE_OS_OTHER -DSQLITE_OMIT_WAL -DSQLITE_OMIT_LOAD_EXTENSION -DSQLITE_THREADSAFE=0 -DSQLITE_OMIT_LOCALTIME -lwasi-emulated-signal -lwasi-emulated-mman`
  - Native: same flags minus SQLITE_OS_OTHER and WASM linker flags, plus `-lm -lpthread`
  - Separate `SQLITE_WASM` and `SQLITE_NATIVE` flag variables for clean separation
- WASM binary size: 916KB after wasm-opt (well under 3MB limit)
- Added parity test in `c-parity.test.ts` with separate skip guard for sqlite3_mem binary
- All 510 WasmVM tests pass, typecheck passes
- Files changed:
  - wasmvm/c/libs/sqlite3/sqlite3.c (new, vendored, 257K lines)
  - wasmvm/c/libs/sqlite3/sqlite3.h (new, vendored)
  - wasmvm/c/libs/sqlite3/LICENSE (new, public domain)
  - wasmvm/c/programs/sqlite3_mem.c (new)
  - wasmvm/c/Makefile (SQLite multi-file override rules)
  - packages/runtime/wasmvm/test/c-parity.test.ts (SQLite parity test)
- **Learnings for future iterations:**
  - `SQLITE_OS_OTHER` requires implementing `sqlite3_os_init()` and `sqlite3_os_end()` stubs — use `#ifdef` to only define them for WASM
  - Native SQLite compilation needs `-lm` (math) and `-lpthread` (even with THREADSAFE=0, the unix VFS references it)
  - SQLite amalgamation compiles to 916KB WASM after wasm-opt — surprisingly compact; validates complex malloc/free/realloc patterns
  - `SQLITE_OMIT_LOCALTIME` avoids references to `localtime_r` which may not work in WASI
  - SQLite 3.46.1 has a typo in `sqlite3IsOverflow` vs `sqlite3IsOVerflow` — not an issue since it only appears with certain flags
