## Codebase Patterns
- CommandRegistry is in packages/kernel/src/command-registry.ts, tests in test/command-registry.test.ts
- TestFileSystem from test/helpers.ts is used for VFS tests (import dynamically in test)
- createMockDriver helper is defined locally in the test file (not from helpers.ts)
- Pre-existing TS error in resource-exhaustion.test.ts (TS18047) and flaky PTY stress test — both on main
- The kernel package typecheck script is `pnpm run typecheck` (not turbo `check-types`)
- MockRuntimeDriver in test/helpers.ts has an optional `tryResolve` property for on-demand discovery tests
- spawnInternal extracts basename before calling tryResolve — drivers always receive the bare command name
- populateBinEntry is async but called fire-and-forget from spawnInternal (sync context); tests need a microtask yield to observe the result
- Standalone binary crate naming: libs use `secureexec-*` (e.g. `secureexec-grep`), command crates use `cmd-*` (e.g. `cmd-grep`)
- Library crates go in `wasmvm/crates/libs/`, binary crates in `wasmvm/crates/commands/`
- Binary crates use `[[bin]] name = "command-name"` and depend on the corresponding library crate
- `cargo build -p cmd-grep --target wasm32-wasip1 -Z build-std=std,panic_abort --release` for WASM builds
- Standalone grep binary is ~864K vs ~15MB multicall — significant size reduction
- WasmVM driver typecheck: `cd packages/runtime/wasmvm && pnpm run typecheck`
- WasmVM driver tests: `cd packages/runtime/wasmvm && pnpm run test`
- Use `node:fs/promises` `open()` (returns FileHandle) for async file operations, NOT separate `read`/`close` imports
- `isWasmBinary()` (async) and `isWasmBinarySync()` are in `packages/runtime/wasmvm/src/wasm-magic.ts`
- Driver in commandDirs mode: commands discovered from filesystem scan, _commandPaths maps name→path
- Driver in legacy mode (wasmBinaryPath only): uses static WASMVM_COMMANDS list, all commands use single binary

- `spawn()` resolves per-command binary path from `_commandPaths` map
