## Codebase Patterns
- Active module-load benchmark artifacts live in `packages/secure-exec/benchmarks/results/module-load/`; refresh them with the full `pnpm --filter secure-exec bench:module-load` run after each performance story.
- Headless CLI workloads inside standalone `NodeRuntime.exec()` should pass `stdin: ""`; for Pi's direct `dist/main.js` print-mode path with an explicit `--print` message, set `process.stdin.isTTY = true` inside the sandbox harness or Pi stalls in its piped-stdin probe before provider traffic.
- Promise-heavy standalone `NodeRuntime.exec()` benchmark scripts need an explicit live handle (for example a short `setInterval()` cleared in `finally`) until the final stdout JSON is emitted, or exec-mode runs can return `code: 0` before long async library work like `pdfDoc.save()` or `zip.generateAsync()` produces output.
- For Pi startup and end-to-end perf stories, record warm wall time together with bridge calls/iteration and dominant IPC counters (`_loadPolyfill`, BridgeResponse bytes, and any new network callsites) instead of relying on a single aggregate number.
- Module-load benchmark progress reporting now comes from generated `packages/secure-exec/benchmarks/results/module-load/comparison.md` plus per-scenario `summary.json`; fixed session overhead is `sample.wallMs - ipc_execute.finish.durationMs` for the matching session in `ipc.ndjson`.
- Module-load phase attribution now comes from `CreateSession -> InjectGlobals -> Execute -> ExecutionResult -> DestroySession` timestamps in per-scenario `ipc.ndjson`, and raw authenticated IPC RTT lives in top-level `transport-rtt.json` / `transport-rtt.md`.
